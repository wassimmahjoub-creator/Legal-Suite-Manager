Implémente l'auto-log silencieux des événements métier sur la timeline d'un dossier juridique.

PRÉREQUIS : T17 livré (cases enrichie + case_opponents + case_team), T17b livré (9 onglets dont "التسلسل الإجرائي" en placeholder).

PHILOSOPHIE :
- AUCUN trigger SQL. Tout passe par un service applicatif explicite.
- Aucun log ne doit faire échouer l'action métier qui l'a déclenché (try/catch silencieux côté logger).
- Tous les événements sont "is_system_generated = true" par défaut, sauf la saisie manuelle.

CREATE TABLE case_events (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  case_id uuid NOT NULL REFERENCES cases(id) ON DELETE CASCADE,
  event_type case_event_type NOT NULL,
  occurred_at timestamptz NOT NULL,
  logged_at timestamptz DEFAULT now(),
  title_ar text NOT NULL,
  title_fr text,
  description text,
  metadata jsonb DEFAULT '{}',
  actor_user_id uuid REFERENCES users(id),
  related_entity_type related_entity_type,
  related_entity_id uuid,
  is_system_generated boolean DEFAULT true,
  case_stage_id uuid, -- FK vers case_stages, à ajouter quand T18b livré
  created_at timestamptz DEFAULT now()
);

CREATE INDEX idx_case_events_case_time ON case_events(case_id, occurred_at DESC);
CREATE INDEX idx_case_events_type ON case_events(event_type);
CREATE INDEX idx_case_events_actor ON case_events(actor_user_id);

class CaseEventLogger {
  static async log({
    caseId,
    eventType,
    occurredAt = new Date(),
    titleAr,
    titleFr = null,
    description = null,
    metadata = {},
    actorUserId,
    relatedEntityType = null,
    relatedEntityId = null,
    isSystemGenerated = true
  }) {
    try {
      const event = await db.case_events.insert({
        case_id: caseId,
        event_type: eventType,
        occurred_at: occurredAt,
        title_ar: titleAr,
        title_fr: titleFr,
        description,
        metadata,
        actor_user_id: actorUserId,
        related_entity_type: relatedEntityType,
        related_entity_id: relatedEntityId,
        is_system_generated: isSystemGenerated
      });

      // Émettre un événement temps réel pour rafraîchir l'UI si possible
      try {
        await pubsub.publish(`case:${caseId}:events`, event);
      } catch (e) {
        // ignore - le log a réussi, c'est l'essentiel
      }

      return event;
    } catch (err) {
      console.error('[CaseEventLogger] Failed to log event', {
        caseId, eventType, error: err.message
      });
      // NE PAS propager l'erreur. L'action métier doit réussir même si le log échoue.
      return null;
    }
  }

ÉTAPE 5 — BACKFILL DES DONNÉES EXISTANTES
═══════════════════════════════════════════════════════

Crée un script de migration scripts/backfill_case_events.ts :

1. Pour chaque case existante sans aucun case_event :
   INSERT case_filed avec occurred_at = COALESCE(opened_at, created_at)
   actor_user_id = case.created_by (si dispo, sinon NULL)

2. Pour chaque case_opponent existant :
   INSERT opponent_added avec occurred_at = case_opponents.created_at

3. Pour chaque case_team existant :
   INSERT team_member_added (et responsible_changed si role = 'responsible')

4. Pour chaque invoice existante :
   INSERT invoice_issued avec occurred_at = invoices.issued_at (ou created_at si NULL)
   SI status = 'paid' : INSERT invoice_paid avec occurred_at = invoices.paid_at OU dernière payment.created_at
   SI status = 'partially_paid' : INSERT invoice_partially_paid

5. Pour chaque payment existant :
   INSERT payment_received avec occurred_at = payments.received_at

6. Pour chaque document avec case_id :
   INSERT document_added avec occurred_at = documents.uploaded_at

Lancer en mode batch nocturne, par chunks de 500 lignes, avec logging du progrès.
