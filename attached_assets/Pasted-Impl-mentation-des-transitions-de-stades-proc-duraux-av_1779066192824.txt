Implémentation des transitions de stades procéduraux avec confirmation utilisateur et création automatique des délais légaux tunisiens.

PRÉREQUIS : T17, T17b, T18a livrés. La table case_events existe, le service CaseEventLogger fonctionne, l'onglet "التسلسل الإجرائي" affiche les événements auto-loggés.

PHILOSOPHIE :
- AUCUNE transition automatique. Le système propose, l'utilisateur confirme TOUJOURS.
- UI simple par défaut. Ne pas ajouter de filtres, options avancées, ou modes complexes.
- Les délais sont calculés selon la matrice fournie, mais TOUJOURS éditables avant validation.
- À chaque action critique, logger un événement via T18a.

ÉTAPE 1 — SCHÉMA DE BASE DE DONNÉES
═══════════════════════════════════════════════════════

Migration 1 : enums et table case_stages

CREATE TYPE case_stage_value AS ENUM ('first_instance', 'appeal', 'cassation', 'execution');
CREATE TYPE decision_outcome AS ENUM ('favorable', 'unfavorable', 'mixed');
CREATE TYPE execution_status AS ENUM ('not_started', 'in_progress', 'completed');

CREATE TABLE case_stages (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  case_id uuid NOT NULL REFERENCES cases(id) ON DELETE CASCADE,
  stage case_stage_value NOT NULL,
  entered_at timestamptz NOT NULL DEFAULT now(),
  exited_at timestamptz,
  court_id uuid REFERENCES courts(id),
  court_case_number text,
  decision_date date,
  decision_summary text,
  decision_outcome decision_outcome,
  decision_document_id uuid REFERENCES documents(id),
  execution_status execution_status,
  execution_notes text,
  notes text,
  created_by uuid REFERENCES users(id),
  created_at timestamptz DEFAULT now()
);

CREATE UNIQUE INDEX idx_case_stages_active 
  ON case_stages(case_id) WHERE exited_at IS NULL;
CREATE INDEX idx_case_stages_case ON case_stages(case_id);

Migration 2 : enum et table legal_deadlines

CREATE TYPE legal_deadline_type AS ENUM (
  'appeal_filing', 'cassation_filing', 'opposition', 
  'execution_notification', 'custom'
);

CREATE TABLE legal_deadlines (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  case_id uuid NOT NULL REFERENCES cases(id) ON DELETE CASCADE,
  case_stage_id uuid REFERENCES case_stages(id) ON DELETE CASCADE,
  deadline_type legal_deadline_type NOT NULL,
  name_ar text NOT NULL,
  start_date date NOT NULL,
  duration_days integer NOT NULL CHECK (duration_days > 0),
  end_date date GENERATED ALWAYS AS (start_date + (duration_days || ' days')::interval)::date STORED,
  reminder_days_before integer DEFAULT 7,
  is_completed boolean DEFAULT false,
  completed_at timestamptz,
  completed_notes text,
  created_at timestamptz DEFAULT now(),
  created_by uuid REFERENCES users(id)
);

CREATE INDEX idx_legal_deadlines_case ON legal_deadlines(case_id);
CREATE INDEX idx_legal_deadlines_stage ON legal_deadlines(case_stage_id);
CREATE INDEX idx_legal_deadlines_end ON legal_deadlines(end_date) WHERE is_completed = false;

Migration 3 : extension de case_events

ALTER TABLE case_events 
  ADD CONSTRAINT fk_case_events_stage 
  FOREIGN KEY (case_stage_id) REFERENCES case_stages(id);

ÉTAPE 2 — INITIALISATION AUTO ET BACKFILL
═══════════════════════════════════════════════════════

Modifier createCase() (T17) :
Après l'insertion en table cases, INSÉRER une ligne case_stages :

INSERT INTO case_stages (case_id, stage, entered_at, court_id, court_case_number, created_by)
VALUES (
  NEW_CASE.id,
  COALESCE(NEW_CASE.litigation_degree, 'first_instance'),
  now(),
  NEW_CASE.court_id,
  NEW_CASE.court_case_number,
  current_user_id
);

Modifier CaseEventLogger.log() :
Avant l'INSERT dans case_events, fetcher le stage actif :
SELECT id FROM case_stages WHERE case_id = ? AND exited_at IS NULL LIMIT 1;
Renseigner case_stage_id dans l'INSERT.

Script de backfill backfill_case_stages.sql :

INSERT INTO case_stages (case_id, stage, entered_at, court_id, created_by)
SELECT 
  c.id,
  COALESCE(c.litigation_degree, 'first_instance'),
  c.created_at,
  c.court_id,
  c.created_by
FROM cases c
WHERE NOT EXISTS (
  SELECT 1 FROM case_stages cs WHERE cs.case_id = c.id
);

-- Renseigner case_stage_id rétroactivement sur case_events existants
UPDATE case_events ce
SET case_stage_id = (
  SELECT cs.id FROM case_stages cs 
  WHERE cs.case_id = ce.case_id 
  AND cs.entered_at <= ce.occurred_at 
  AND (cs.exited_at IS NULL OR cs.exited_at >= ce.occurred_at)
  ORDER BY cs.entered_at DESC LIMIT 1
)
WHERE ce.case_stage_id IS NULL;

ACTION À LA CONFIRMATION :

Transaction DB (rollback si échec) :

1. UPDATE case_stages SET 
     exited_at = now(),
     decision_date = ?,
     decision_summary = ?,
     decision_outcome = ?,
     decision_document_id = ?
   WHERE id = current_stage.id;

2. INSERT INTO case_stages (case_id, stage, entered_at, court_id, court_case_number, created_by)
   VALUES (?, next_stage, now(), ?, ?, current_user);

3. Pour chaque délai dans section 3 :
   INSERT INTO legal_deadlines (case_id, case_stage_id, deadline_type, name_ar, start_date, duration_days, reminder_days_before, created_by)
   VALUES (?, new_stage.id, ?, ?, ?, ?, ?, current_user);

4. UPDATE cases SET litigation_degree = next_stage WHERE id = ?;

5. Logger via CaseEventLogger (T18a) :
   - event 'judgment_recorded' sur l'ancien stage avec metadata { outcome, summary }
   - event 'stage_transitioned' sur le nouveau stage avec metadata { from_stage_ar, to_stage_ar, outcome }
   - event 'legal_deadline_added' pour CHAQUE délai créé

6. Toast succès "تم الانتقال إلى الطور [next_stage_ar]"

7. Refresh : stepper + timeline + onglet "الحكم والتنفيذ"
